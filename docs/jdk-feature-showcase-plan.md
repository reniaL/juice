# JDK 版本特性演示项目总体方案

## 1. 项目定位

本项目用于系统化说明和演示 JDK 8、11、17、21、25 这些代表性版本中的重要新特性、新语法与关键 API 变化。

项目名称采用当前仓库目录名称：juice。

项目的核心目标不是罗列全部发布说明，而是围绕"开发中值得掌握、适合演示、便于迁移理解"的内容，提供：

- 可直接运行的示例代码
- 配套的 Markdown 讲解文章
- 清晰的版本演进脉络

最终希望让读者既能按版本学习，也能在实际开发和版本升级时快速查阅。

## 2. 项目目标

### 2.1 主要目标

1. 在一个聚合工程中按 JDK 版本拆分代码模块，集中展示各目标版本的代表性能力。
2. 为每个版本编写一篇结构统一的 Markdown 文章，说明背景、语法、场景与示例。
3. 让示例代码尽量短小、明确、可运行，避免被业务逻辑淹没。
4. 建立一套统一的写作和代码组织规范，便于后续持续补充。
5. 形成适合学习、分享、培训和版本迁移参考的资料仓库。

### 2.2 非目标

1. 不追求覆盖每个小版本的全部更新项。
2. 不以 JVM 调优、GC 深入原理、JIT 内部实现为第一阶段重点。
3. 不优先收录仅适合特定框架或大型工程背景才能理解的示例。
4. 默认不把预览特性和孵化特性作为正文主线内容，除非该版本专题明确需要扩展说明。

## 3. 面向读者

- 想快速了解 Java 版本演进的开发者
- 正在做 JDK 升级评估或技术分享的工程师
- 需要整理 Java 新语法与新特性的学习者
- 希望通过示例代码理解 API 变化而不是只看文字说明的读者

## 4. 版本范围

本项目第一阶段覆盖以下版本：

1. JDK 8
2. JDK 11
3. JDK 17
4. JDK 21
5. JDK 25

这些版本具有明确的代表性：

- JDK 8 是现代 Java 编程风格的重要分水岭。
- JDK 11 是长期支持版本，适合作为从旧版本迁移的关键节点。
- JDK 17 是新一代主流 LTS 版本，语言特性变化明显。
- JDK 21 是当前主流 LTS 版本之一，并发与模式匹配能力值得重点展示。
- JDK 25 可作为后续阶段的最新 LTS 版本专题，体现当前 Java 演进方向。

## 5. 内容选择原则

每个版本收录的内容优先满足以下条件：

1. 对日常开发有明显帮助。
2. 能体现语言风格、代码组织方式或标准库使用方式的变化。
3. 能用简洁示例准确表达，不依赖复杂业务上下文。
4. 适合和旧写法形成对比，便于读者理解升级价值。
5. 优先选择正式特性；预览或孵化能力单独标注。

## 6. 推荐项目结构

当前阶段更推荐采用"文档集中 + 代码分版本模块化"的结构。具体做法是：在根目录维护统一的文档、导航和聚合构建，在代码层为每个目标 JDK 版本提供一个轻量子模块。

这样做的好处是：

1. 每个版本都可以配置自己的编译选项，例如 release、预览特性或工具链要求。
2. 示例代码的版本归属是可验证的，而不只是通过包名约定。
3. 文档仍然集中在根目录的 docs 中，阅读体验不会因为模块拆分而变差。
4. 结构虽然是多模块，但每个模块可以保持很轻，不会形成过度设计。

需要明确的边界是：

1. 这里的多模块目标是版本隔离与示例验证，不是把工程设计成复杂平台项目。
2. 除非后续确实出现大量共享代码，否则不必急着抽出 common 模块。
3. 每个版本模块只保留该版本最值得展示的示例，避免模块数量增加后内容失控。

也就是说，第一阶段采用多模块，但保持轻量；让复杂度只服务于"按版本正确编译和演示"这个核心目标。

## 7. 代码组织建议

不建议把所有示例堆到一个类中，也不建议每个模块只放一个巨大入口类。更合适的方式是：按版本拆模块，在模块内部再按特性分包。

建议结构：

```text
jdk-17-demo/
└─ src/main/java/
   └─ com/juice/jdk17/
      ├─ records/
      ├─ sealed/
      ├─ patternmatching/
      ├─ textblocks/
      └─ DemoApplication.java
```

建议约定：

1. 每个版本对应一个独立模块。
2. 每个模块内部使用独立顶层包，例如 com.juice.jdk8、com.juice.jdk17。
3. 每个特性一个子包。
4. 每个子包包含 1 到 3 个最小示例类。
5. 每个模块保留一个少量入口类，用于快速运行该版本的主要示例。
6. 如有必要，为关键特性补充对比类，例如 OldStyleExample 与 NewStyleExample。

## 8. 每篇文章的统一结构

为保证后续内容风格一致，建议每篇版本文章都遵循同一模板。

### 8.1 建议模板

1. 版本简介
2. 为什么这个版本值得关注
3. 本文覆盖的主要特性
4. 各特性详解
5. 与旧版本写法对比
6. 适用场景与落地建议
7. 升级或使用注意事项
8. 本章小结

### 8.2 单个特性的讲解模板

每个特性建议按以下结构展开：

1. 背景与痛点
2. 核心语法或 API 说明
3. 最小可运行示例
4. 示例代码解析
5. 典型使用场景
6. 常见误区或限制
7. 与旧写法对比

## 9. 各版本选题建议

以下内容用于锁定第一阶段的讲解范围，后续每个版本专题可在此基础上细化。

### 9.1 JDK 8

建议优先覆盖：

1. Lambda 表达式
2. 函数式接口
3. Stream API
4. Optional
5. 接口默认方法与静态方法
6. 新日期时间 API
7. CompletableFuture

说明：
JDK 8 是整个项目的基础章节，重点应放在编程范式变化上，而不只是语法糖本身。

### 9.2 JDK 11

建议优先覆盖：

1. HttpClient API
2. 单文件源码运行
3. String API 增强
4. Files API 增强
5. var 在 Lambda 参数中的使用
6. Optional 与集合相关 API 的补充能力

说明：
JDK 11 的亮点不只在语法，而在于标准库可用性增强，因此文档中应强调实用价值。

### 9.3 JDK 17

建议优先覆盖：

1. Record
2. Sealed Class
3. Pattern Matching for instanceof
4. Text Blocks
5. 改进的空指针异常信息

说明：
JDK 17 适合突出"建模能力"和"代码可读性"的提升。

### 9.4 JDK 21

建议优先覆盖：

1. Virtual Threads
2. Record Patterns
3. Pattern Matching for switch
4. Sequenced Collections
5. Scoped Values 或其他并发相关扩展内容（如需扩展时单独标注稳定性）

说明：
JDK 21 是并发模型与模式匹配的重要里程碑，应重点展示其对代码组织与吞吐能力的影响。

### 9.5 JDK 25

建议策略：

1. 先为 JDK 25 预留独立模块与专题文章。
2. 选题以该版本正式发布后的稳定特性为准。
3. 若存在预览或孵化能力，只放在扩展阅读，不作为第一阶段主线内容。

说明：
由于项目当前处于总体规划阶段，JDK 25 的专题在正式展开时应再次核对 GA 内容，避免把非正式能力混入主线示例。

## 10. 示例代码编写规范

为了保证项目可读性和可维护性，建议采用以下约定：

1. 示例代码优先短小、单一职责、可独立理解。
2. 一个示例只表达一个核心知识点。
3. 尽量同时给出旧写法与新写法的对照。
4. 示例类命名直接表达主题，例如 LambdaBasicExample、RecordDemo、VirtualThreadDemo。
5. 避免引入与特性无关的第三方框架。
6. 需要输出结果时，优先使用简单的控制台输出或断言。

## 11. 构建与运行建议

当前阶段建议以 Maven 聚合工程为主，在父工程下管理多个按版本划分的轻量子模块。这样既能保持统一入口，又能保证不同 JDK 版本使用各自的编译配置。

建议策略：

1. 根项目使用父 pom.xml 聚合所有版本模块。
2. 每个子模块单独配置对应的 maven.compiler.release。
3. 示例优先支持通过 main 方法直接运行。
4. 对适合断言验证的示例，可补充少量测试代码。
5. 在 README 中提供按版本构建、运行和阅读的说明。
6. 如需演示预览特性，可在对应模块中单独增加 profile 或插件参数，而不影响其他模块。

## 12. 输出成果设计

第一阶段的成果建议包括：

1. 一个可构建的 Maven 聚合工程骨架。
2. 五个按 JDK 版本划分的轻量代码模块。
3. 一篇总览文档。
4. 五篇版本专题文档的初稿模板。
5. 每个版本至少 3 到 5 个代表性示例。

## 13. 推荐实施顺序

建议按以下顺序推进，而不是同时铺开全部内容：

1. 先搭建 juice 的聚合工程骨架与基础目录结构。
2. 优先完成 JDK 8 专题，作为整个项目的写法基准。
3. 再补 JDK 11 和 JDK 17，建立主流迁移路径。
4. 之后完成 JDK 21，突出最新主流特性。
5. 最后整理 JDK 25，保持内容与正式版本同步。
6. 如果后续出现共享代码较多的情况，再评估是否引入 common 模块。

## 14. 后续迭代建议

当第一阶段完成后，可以继续增加以下内容：

1. 按主题阅读的导航页，例如并发、语法、类型系统、标准库。
2. 版本迁移指南，例如从 8 升到 17、从 11 升到 21。
3. 示例对比图表，例如"旧写法 vs 新写法"。
4. 与实际工程实践结合的章节，例如哪些特性适合在业务代码中优先落地。
5. 对个别需要特殊编译参数的版本特性，在对应版本模块内增加实验性 profile 或单独示例分支。

## 15. 当前结论

该项目当前更适合采用"文档集中在 docs + 代码按 JDK 版本拆分为轻量模块 + 按版本编写 Markdown 专题"的方式推进。

这个方案兼顾了两件事：一是读者能在统一文档入口中快速浏览内容，二是每个版本的示例代码都可以用对应编译配置进行验证。对于 juice 这样的学习型仓库，这是一种更稳妥的基础结构。
